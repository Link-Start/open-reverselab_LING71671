# Database Dump Cleaning — 数据库泄露数据清洗方法论

> 从非结构化或半结构化泄露数据中提取、分类、关联有效信息，建立源数据到业务实体的映射。

## 关键词

`数据清洗` `泄露数据` `格式分类` `特征提取` `锚点映射` `置信度分级` `数据去噪` `前缀聚类`

## 0. 流程全景

```
原始数据 → 结构识别 → 有效载荷提取 → 特征聚类 → 锚点关联 → 去噪验证 → 结构化输出
  │           │            │            │          │          │           │
HTML/JSON  定位数据块   去除标记行   长度/字符集  样本→实体   假数据剔除   CSV/DB
纯文本/CSV  识别分隔符   去重        前缀聚类     格式→名称   不可用过滤   索引
```

## 1. 源数据结构识别

### 1.1 原始格式判定

| 格式 | 特征 | 提取策略 |
|------|------|---------|
| HTML 堆叠 | 多段 `<body>` 标签、多个 `<textarea>`/`<table>` | 定位目标标签，提取 textContent |
| JSON 响应 | `{...}` 或 `[...]` 结构 | 解析后遍历字段路径 |
| 纯文本列表 | 每行一条记录，分隔符为换行 | 按行处理，识别行类型 |
| 混合格式 | 数字行+文本行+标记行交错 | 逐行分类，建立行类型状态机 |

### 1.2 行类型分类

```
数据行:    符合预期格式（长度/字符集/前缀）
分隔符行:  短数字（如 "1"）、空行
标记行:    含特定关键词（状态/标签）
Header行:  文件开头几行，数值或元数据
冗余数据:  与其他行重复
```

### 1.3 数据块定位

有效数据常被包裹在噪声中：
- 错误页输出（校验失败但代码继续执行）
- 标签/标记行穿插
- 重复的 header 信息

策略：**状态机扫描** — 定义进入/退出条件，仅提取数据块内的行。

## 2. 特征维度

### 2.1 基础维度

| 维度 | 用途 |
|------|------|
| 长度分布 | 聚类第一特征，同格式往往等长 |
| 字符集 | `[A-Z0-9]` vs `[a-f0-9]` vs 含特殊字符 |
| 前缀（2-4 字符） | 最强区分特征 |
| 结构模式 | 固定段 + 可变段 + 分隔符的模式匹配 |

### 2.2 前缀聚类

```python
# 前缀分组伪代码
groups = {}
for record in data:
    prefix = extract_prefix(record, length=2)  # 先 2 字符
    if is_counter_variant(prefix):             # 计数器子类合并
        prefix = normalize(prefix)
    groups[prefix].append(record)
```

### 2.3 计数器归一化

许多格式的前缀第三位是十六进制计数器（`TK0`-`TKF`），实际为同一类型。
- 识别模式：前缀 + `[0-9A-F]` + 固定段
- 合并策略：计数器位归一化为通配符

## 3. 可独立使用资产识别

### 3.1 可直接使用

| 类型 | 特征 | 判定方式 |
|------|------|---------|
| URL/下载链接 | `https?://` 开头 | 正则匹配 |
| 邮箱+密码 | `@` 符号 + `----` 分隔符 | 结构识别 |
| 手机号+API | 国际区号 + 固定长度 + URL | 模式匹配 |
| 联系方式 | 特定前缀 + 可识别 ID | 关键词匹配 |

### 3.2 需上下文才能使用

| 类型 | 特征 | 原因 |
|------|------|------|
| 短激活码 | 6-12 字符，纯字母数字 | 不知道对应哪个产品/平台 |
| UUID 格式码 | 标准 UUID 结构 | 同上 |
| 纯数字 | 8+ 位数字 | 可能是订单号/充值卡/假数据 |

### 3.3 假数据特征

- 重复模式（`123456`、`111111`、`asd123`）
- 明显占位符（`dasdasdasd`、`rdtv`）
- 不符合任何已知业务格式

## 4. 锚点映射

### 4.1 原理

从有限的 ground truth 样本出发，通过格式一致性将映射关系扩展到同格式的全部数据。

```
样本:  { CDK_prefix_A → Product_X }  (来源: 订单/API等可信数据)
扩展:  所有 prefix_A 的 CDK → Product_X
```

### 4.2 锚点来源

| 来源 | 可信度 | 获取难度 |
|------|--------|---------|
| 订单/交易记录 | 最高 | 需要 IDOR 或 API 访问 |
| 商品描述匹配 | 高 | 需要商品目录 |
| 品牌名拼音推断 | 中 | 需要商品目录 + 拼音映射表 |
| 格式结构推测 | 低 | 仅需泄露数据本身 |

### 4.3 扩展验证

锚点质量通过以下方式校验：
- 同一前缀下的数据格式是否完全一致
- 同一商品下不同锚点返回的前缀是否一致
- 格式块大小与商品库存量是否在同一数量级

## 5. 品牌/名称关联

### 5.1 直接匹配

泄露数据的前缀出现在商品名称的显著位置（如品牌名被 `【】` 包裹）。

```
前缀 "CA" → 商品名中的 "【Canary】" → 匹配
```

### 5.2 拼音首字母

对中文品牌名逐字取拼音首字母，匹配泄露数据前缀。

```
品牌 "雷豹" → 拼音首字母 "LB" → 匹配前缀 "LB"
品牌 "星穹" → 拼音首字母 "XQ" → 匹配前缀 "XQ"
```

需要构建中文→拼音首字母的映射表，处理多音字和生僻字。

### 5.3 英文缩写

```
"PUBG" → 前缀 "GU"、"PU"
"Minecraft" → 前缀 "MC"
```

### 5.4 收敛策略

无法通过以上方式匹配的前缀：
- 数量 < 阈值 → 归入"未分类"
- 结构可识别 → 标记格式类型（如"32位 hex CDK"）
- 无任何特征 → 标记为"未知"

## 6. 置信度分级

| 级别 | 标准 | 示例 |
|------|------|------|
| **确认** | 有订单/API 直接证据 | 订单返回 `CDK_X + tid_Y` |
| **高** | 品牌名直接匹配 | 前缀 = 商品名中的品牌 |
| **中** | 拼音/缩写匹配 + 格式一致 | 前缀 = 品牌拼音首字母，且格式统一 |
| **低** | 纯格式推断 | 仅按长度/字符集归类 |

## 7. 去噪

### 7.1 假数据识别

- 重复值聚类 — 异常高频的同一内容
- 模式匹配 — 连续数字、键盘序列、常见占位符
- 长度异常 — 与其他数据分布显著偏离

### 7.2 业务验证

- 有效凭证格式：符合已知的卡密/激活码结构
- 有效账号格式：符合邮箱/手机号格式规范
- 来源交叉：来自可信源（如 API）的数据优先保留

## 8. 输出规范

清洗完成后输出应包含：

```
每条记录:
  - 原始行号（可追溯）
  - 数据内容（保留原始）
  - 分类标签（数据类型）
  - 关联实体 ID（如商品 ID）
  - 关联实体名称（如商品名）
  - 置信度
```

输出格式：CSV（全量）、SQLite（查询）、HTML（浏览）。
